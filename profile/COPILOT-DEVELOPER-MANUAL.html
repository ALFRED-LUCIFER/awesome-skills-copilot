The Copilot Agent System Copilot Agent System is an AI-powered engineering platform shared across all repositories in the Org GitHub Enterprise organization. It provides specialised agents for planning, building, testing, reviewing, and documenting software — all enforced by mandatory quality chains and security guardrails.

A.1 Prerequisites

  • VS Code — latest stable version
  • GitHub Copilot extension — version ≥ 1.250 with Copilot Chat enabled
  • GitHub Enterprise account — member of the Org organization
  • MCP environment variables — see section H.3 for Jira/Confluence integration

A.2 Install the VS Code Copilot plugin (recommended)

From v2.0.0, this repo is packaged as the private-copilot VS Code Copilot plugin (manifest at .github/plugin/plugin.json). Install registers all 14 Avenger-themed agents, 26 skills, and 4 hooks in one step.

  1. VS Code → Command Palette → Chat: Install Plugin From Source.
  2. Paste the Git URL of .github-private.
  3. Reload VS Code when prompted.
  4. Add one line to user settings.json so the 14 prompts register: 
    "chat.promptFilesLocations": {
  "${userHome}/.vscode/extensions/private-copilot-*/prompts": true
}

Validate before pushing changes to the plugin: npm run validate (filename ↔ name: parity, plugin.json schema, skill manifests).

A.2-Legacy · Clone & symlink (pre-plugin workflow, still supported)

Agents are also shared organization-wide via GitHub enterprise REST discovery. The clone-and-symlink path remains supported for users who prefer not to install the plugin.

Step 1 — Clone this repository

git clone https://myorg-com.ghe.com/myorg/.github-private.git ~/myorg-copilot-config

Step 2 — Symlink prompts and skills to your user folder

This makes all slash commands and skills available in every workspace, even without org-level sharing:

# macOS / Linux
ln -s ~/myorg-copilot-config/prompts ~/.vscode/prompts
ln -s ~/myorg-copilot-config/skills  ~/.vscode/skills

# Windows (PowerShell as Admin)
New-Item -ItemType SymbolicLink -Path "$env:USERPROFILE\.vscode\prompts" -Target "$env:USERPROFILE\myorg-copilot-config\prompts"
New-Item -ItemType SymbolicLink -Path "$env:USERPROFILE\.vscode\skills"  -Target "$env:USERPROFILE\myorg-copilot-config\skills"

Step 3 — Update VS Code settings

Add the following to your User Settings (Cmd+, → Open Settings JSON):

{
  "github.copilot.chat.codeGeneration.useInstructionFiles": true,
  "chat.promptFiles": true,
  "chat.promptFilesLocations": {
    ".github/prompts": true,
    "~/.vscode/prompts": true
  },
  "github.copilot.chat.codesearch.enabled": true,
  "chat.mcp.access": "all"
}
Organization sharing: Agents (agents/) are fully shared at the organization level — they appear for all members automatically. Prompts and skills are also org-shared, but symlinking gives you local access for offline use and faster loading.

Multi-Model Panel Composition (--full / --panel mode)

When --full is triggered, four specialist sub-reviewers run in parallel, then cross-examine each other's findings before the Coordinator synthesises the final verdict.

RoleModelResponsibilityBias
🔒 Security AuditorGPT-5.5OWASP Top 10, SEC-1–24, auth, XSS, IDOR, secrets, injectionStrict — flags aggressively
🏗 Architecture JudgeClaude Opus 4.6Layer violations, SOLID, DDD, controller pattern, coupling, YAGNIConservative — questions necessity
✅ Quality EnforcerClaude Sonnet 4.5DUP-1–5, WCAG 2.2 AA, readability, test coverage, naming, i18nThorough — exhaustive checklist
🧩 Codex Code AgentGPT-5.3-CodexCompile/type correctness, API misuse, regression risk, patch feasibilityPragmatic — validates fixability
🎯 CoordinatorClaude Sonnet 4.6Synthesises panel, resolves conflicts, computes final score, drives auto-fixBalanced — resolves disagreements

Panel Workflow

STEP 1 — PARALLEL DEEP SCAN (all 4 specialists run simultaneously) ├── Security Auditor → SEC-1–24, OWASP, auth, secrets, injection ├── Architecture Judge → Layer violations, coupling, SOLID, DDD, YAGNI ├── Quality Enforcer → DUP-1–5, WCAG 2.2 AA, readability, test coverage, naming └── Codex Code Agent → Compile/type failures, API misuse, patch feasibility STEP 2 — CROSS-EXAMINATION (specialists challenge each other) Each panelist reviews the other reports and replies: ├── CONFIRM — "I agree — here is supporting evidence" ├── ESCALATE — "This is worse — severity should be 🔴 because…" ├── DISPUTE — "I disagree — this is intentional because…" └── ADD — "Related issue the other panelist missed: [finding]" STEP 3 — COORDINATOR SYNTHESIS Resolves disputes → deduplicates overlaps → computes final score Produces ranked list: Critical → High → Medium → Low STEP 4 — AUTO-FIX LOOP (if score > 5) Delegate fixes to @backend / @frontend → re-run panel → max 3 iterations

Scoring

SeverityScoreExample
🔴 Critical+10SQL injection, hardcoded secret, missing auth
🟠 High+5N+1 query, SOLID violation, missing error handling
🟡 Medium+2Hardcoded UI string not in i18n, missing aria-label
🔵 Low+1Naming convention, minor duplication
Pass threshold: score ≤ 5. Score > 5 activates the auto-fix loop — fixes are delegated to @frontend or @backend, then re-reviewed. Maximum 3 iterations before escalating to you.

Conflict Resolution Between Panelists

ConflictTiebreaker Rule
Security Auditor says 🔴, others say 🟡Security wins — never downgraded without explicit evidence
Architecture Judge says 🟠, Security says 🔵Security wins for any finding with an attack vector
Quality Enforcer says 🔴, others disagreeCoordinator decides — Quality rarely has Critical; assess user impact
Codex says a fix cannot compileCodex blocks auto-fix — require compile-safe patch first
Two panelists agree, one disputesMajority rules — disputed panelist must provide new evidence
All panelists disagreeCoordinator casts deciding vote — must cite specific GUARDRAILS rule

Auto-Fix Loop Example

@reviewer --full src/domain/order/ → Iteration 1: Full panel → score 22 Delegate: @backend (8 issues) + @frontend (4 issues) → Iteration 2: Full panel on changed files → score 8 Delegate: remaining 3 issues → Iteration 3: Full panel on changed files → score 3 ✅ PASS
Performance tip: The full panel is thorough but slower. Use --quality (default) for routine reviews and --full only for pre-PR gates and mandatory chain invocations.

F.2 Memory & Context Retention

The agent system uses a two-layer memory architecture to reduce context loss between sessions and share learnings across all projects in the organization.

Why this matters: Without memory, every session starts from zero. Developers re-discover the same gotchas, repeat the same architectural decisions, and lose planning context when a conversation ends. The memory system solves this org-wide.

F.2a Two-Layer Architecture

LayerLocationScopeLifetimeUpdated By
Org-level.github-private/memories/org/All projectsPermanent (git)Developer PR
Repo-level.copilot/memories/repo/Single project30 days (auto-pruned)session-logger hook
Session/memories/session/Current conversationConversation onlyAgent (Copilot-native)

memories/org/anti-patterns.mdKnown bad patterns — promoted from repo-level gotchas when recurring
memories/org/architecture-decisions.mdOrg-wide ADRs (e.g., "why shared agent system", "why two-layer memory")

Repo-Level Memory (Auto-Generated)

F.2b Session Hooks for Context

Session Start (log-session-start.sh)

  • Logs session start event to logs/copilot/session.log
  • Prints last session summary path — developer sees continuity hint
  • Shows accumulated gotcha count as a reminder

Session End (log-session-end.sh)

  • Generates session summary to .copilot/memories/repo/session-summaries/YYYY-MM-DD-HHmm.md
  • Captures gotchas via expanded keyword scan: gotcha:, lesson:, remember:, important:, note for future:, trap:, pitfall:, watch out:, careful:, beware:
  • Prunes session summaries older than 30 days
  • Suggests org-level promotion when >5 gotchas accumulate

F.2c Best Practices

✅ DO
  • Use keywords (gotcha:, lesson:, remember:) in conversations for auto-capture
  • Ask Copilot to "check my last session summary" when resuming work
  • Review discovered-gotchas.md periodically — promote recurring ones to org-level via PR
  • Write to /memories/session/ for complex multi-step tasks within a session
  • Check org memories before making architecture decisions
❌ DON'T
  • Don't expect full conversation replay — hooks capture metadata, not full text
  • Don't modify org-level memories directly — always via PR to .github-private
  • Don't store secrets in memory files — they're committed to git
  • Don't ignore the promotion hints — accumulated gotchas signal systemic issues
Instruction file: The memory-conventions.instructions.md (applyTo: **) auto-loads for all agents, ensuring they follow these memory rules without manual invocation.

F.3 Knowledge Base

The knowledge base is a set of living documents that give LLMs instant domain context — without exhausting the context window or forcing the agent to re-read source files from scratch. It complements the memory system: memory tracks what happened in past sessions; the knowledge base captures what the code does.

Why this matters: When an agent starts working in a large codebase, it must spend tokens exploring source files just to understand the domain. A pre-built knowledge base short-circuits that exploration — the agent reads one AGENTS.md and has everything it needs to proceed.

What Gets Created

FilePurposeUpdated By
{folder}/AGENTS.md1-page index per domain folder — purpose, key entities, entry points, key files, links. VS Code Copilot reads this automatically when working in that folder./knowledge-init
docs/{domain}.mdDeep knowledge doc — bounded context, data model, business rules, flows, external contracts, known gotchas./knowledge-init
docs/INDEX.mdMaster index — all domains, optional project metadata (Value Stream, Confluence URL, Jira project keys, Jira epics)./knowledge-init
.copilot/memories/repo/knowledge-drift.mdStale domain log — entries written automatically by the knowledge-drift hook when source files change.knowledge-drift hook

How to Build It

# First run — full scan (prompts for optional project metadata)
/knowledge-init

# Scope to one domain
/knowledge-init Orders

# Refresh stale domains flagged by knowledge-drift hook
/knowledge-init --update

# Update project metadata only
/knowledge-init --meta-only

Automatic Drift Detection

The knowledge-drift hook runs at every session end. It detects source files changed during the session, maps them to their domain folder, and appends a STALE: entry to the drift log. On your next session, Copilot will surface these stale domains. Run /knowledge-init --update to refresh only the affected domains.

Update safety: Sections you annotate with <!-- manual --> in any knowledge doc are never overwritten by --update. Use this for hand-written business rules and gotchas that require human judgment.

G.1g Knowledge Base Skill (1)

knowledge-init

Builds and maintains a living knowledge base for any application repo — domain by domain. App/repo level, not tied to any specific tech stack.

Produces:

  • {folder}/AGENTS.md — 1-page memory-map index per domain folder (VS Code Copilot reads this automatically when working in that folder). Contains: purpose, key entities, entry points, key files, links to deep docs.

Scope: Domain knowledge only — what the code does. Coding standards live in GUARDRAILS. Test patterns live in testing-standards instructions.

Used by: /knowledge-init

G.3 Hooks

Hooks are Copilot Coding Agent lifecycle extensions in hooks/. They run automatically during coding agent sessions to enforce governance, security, auditability, and knowledge freshness — no developer configuration required.

HookTriggerPurpose
governance-auditAgent prompt submissionScans agent prompts for threat signals (prompt injection, jailbreaks) and logs governance events
knowledge-driftSession endDetects source files changed during the session, maps them to domain folders, and appends STALE: entries to .copilot/memories/repo/knowledge-drift.md — so /knowledge-init --update knows what to refresh next run
secrets-scannerFile modificationDetects leaked secrets, credentials, API keys, and sensitive data in files modified during agent sessions
session-loggerSession lifecycleLogs all Copilot coding agent activity for audit trail and post-session analysis
tool-guardianTool invocationBlocks dangerous ops (rm -rf, git push --force, DROP TABLE) before the agent executes them

Benefits

  • Security: secrets-scanner + tool-guardian prevent accidental credential leaks and destructive operations
  • Compliance: governance-audit + session-logger provide ISO 42001 audit trail for AI-assisted development
  • Safety: tool-guardian enforces GUARDRAILS § P5 (No Destructive Commands) at the tool level
  • Knowledge freshness: knowledge-drift automatically flags stale AGENTS.md files when source code changes — keeping the knowledge base current without manual effort
  • Zero configuration: Hooks activate automatically when the plugin is installed

How Hooks Run

  1. Hooks are declared in plugin.json under the hooks field
  2. VS Code loads them when the plugin is installed — no manual activation needed
  3. Each hook has a hook.md file defining its trigger event and logic
  4. Hooks execute in the coding agent's lifecycle (before tool calls, after file edits, on session start/end)
  5. Violations are logged and optionally block the operation (e.g., tool-guardian blocks destructive commands)

# MCP server headers

export CONFLUENCE_MCP_API_KEY="your-confluence-mcp-api-key"
